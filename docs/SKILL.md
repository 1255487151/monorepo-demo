# VitePress 文档编写规范

## 1. 目标

本规范用于统一当前仓库 `docs` 目录下 VitePress 文档站的写法、结构和维护方式。

目标不是记录零散约定，而是明确：

- 文档应该写到哪里
- 每类页面应该怎么写
- 哪些内容可以写，哪些内容不能写
- 页面更新时要遵守什么标准

本规范适用于以下目录：

- `docs/.vitepress`
- `docs/index.md`
- `docs/guide`
- `docs/packages`
- `docs/learn-config`
- `docs/specs`

## 2. 站点级规范

### 2.1 主题规范

- 站点默认暗色
- 必须保留主题切换能力
- 整体视觉参考 Element Plus 文档站的阅读秩序，但不复刻其浅色方案
- 风格要求：清晰、结构化、API 优先、可快速定位

### 2.2 导航规范

文档主导航固定为以下四类：

- 首页 / 入门
- 库文档
- 配置学习
- 规范要求

禁止新增与这四类定位重复的平行导航分组。

### 2.3 语言规范

- 文档默认使用中文
- 标题命名应简洁、稳定、语义清晰
- 文案优先描述当前仓库真实实现，不写空泛宣传语

## 3. 信息架构规范

### 3.1 首页 / 入门

首页和入门页只负责：

- 项目定位
- 文档阅读路径
- 发布库边界
- monorepo 关系说明

禁止事项：

- 禁止在首页堆 API 表格
- 禁止在首页堆长篇配置解释
- 禁止在首页堆规范全文

### 3.2 库文档

库文档只承接当前对外发布的库。

当前固定范围：

- `@smallbrother/components`
- `@smallbrother/directives`
- `@smallbrother/utils`

`packages/playground` 不属于发布库文档范围。

### 3.3 配置学习

配置类文档统一放在 `docs/learn-config/*`，当前包括：

- `eslint.md`
- `git.md`
- `tsconfig.md`
- `vite-config.md`

配置学习页必须直接承接原始配置文档中仍然有效的正文内容，不能只写摘要。

### 3.4 规范要求

规范、要求、约束、验收标准类文档统一放在 `docs/specs/*`。

当前包括：

- `table-skill.md`

规范页只负责规则和边界，不负责教学式配置说明，也不负责库 API 介绍。

## 4. 页面类型写法规范

### 4.1 Components 文档写法

`components` 必须使用“库入口页 + 组件子页”的组织方式：

- `docs/packages/components/index.md`
- `docs/packages/components/select.md`
- `docs/packages/components/table.md`

禁止把 `components` 写回单页介绍文档。

#### 组件库入口页必须包含

- 安装方式
- 导入方式
- 组件总览
- 样式入口
- 导出入口
- 类型入口
- 内部依赖

#### 组件子页必须包含

- 安装 / 导入
- 基础用法
- 示例代码块
- Props / Attributes 表
- Slots 表
- Events 表
- Exposes / Methods 表
- 类型说明
- 样式与依赖说明

#### 组件页写法要求

- 写法接近 Element Plus 组件文档
- 重点展示“怎么用”和“接口是什么”
- 示例代码必须贴近当前仓库真实导出与真实实现
- Props、Slots、Events、Exposes 必须表格化
- 不把规范要求混入组件页
- 不编造源码中不存在的属性、事件、方法和插槽

### 4.2 Directives 文档写法

`directives` 当前保持聚合页，但内容必须 API 化。

页面至少包含：

- 安装方式
- 导出说明
- 注册方式
- API
- 参数或绑定值表
- 示例
- 行为说明
- 限制
- 依赖与产物

禁止写成纯库介绍页。

### 4.3 Utils 文档写法

`utils` 当前保持聚合页，但必须围绕每个导出方法组织内容。

页面至少包含：

- 安装方式
- 导出说明
- 使用方式
- 每个函数的签名
- 参数表
- 返回值或行为说明
- 示例
- 限制
- 依赖与产物

禁止只写“工具库介绍”，不写 API。

### 4.4 配置学习页写法

每个配置学习页必须包含以下章节中的必要部分：

- 当前作用
- 对应源码配置
- 关键理解点
- 当前仓库中的实际配置片段
- 在当前仓库里的使用建议

写法要求：

- 要吸收原始 md 文档中仍然有效的正文和示例
- 要删除和当前仓库无关的百科式内容
- 要从“通用教程”收口到“当前仓库怎么用”

禁止事项：

- 禁止只做摘要转述
- 禁止完全照搬旧文档不做重组
- 禁止把配置学习页写成规范页

### 4.5 规范要求页写法

每个规范页必须包含以下章节中的必要部分：

- 目标与当前状态
- 组件或能力定位
- 必须保留的能力
- 实现规则
- 禁止事项
- 验收标准
- 迁移原则

写法要求：

- 直接说明规则
- 使用可执行、可评审的表达
- 不与库文档、配置学习页混写

## 5. 内容来源优先级

文档内容来源优先级必须固定如下：

1. 当前仓库源码
2. `package.json`、导出配置、构建配置
3. 包 README
4. 历史文档中的有效正文

若历史文档与当前源码不一致，以当前源码为准。

## 6. 迁移与删除规范

当旧文档内容已经迁入新的 VitePress 页面后：

- 必须删除旧的根层文档
- 不保留重复内容
- 不保留仅用于兼容的空壳页面

当前已经纳入该规则的文件包括：

- `docs/ESLINT.md`
- `docs/GIT.md`
- `docs/TSCONFIG.md`
- `docs/VITE_CONFIG.md`
- `docs/table-skill.md`

后续如有类似旧文档迁移，也遵循同一规则。

## 7. 更新规范

出现以下变化时，必须同步更新文档：

- 发布库新增或删除公开导出
- 组件 Props / Slots / Events / Exposes 变化
- 指令参数、行为或注册方式变化
- 工具函数签名或行为变化
- 配置入口、脚本命令、构建行为变化
- 规范要求变化

## 8. 禁止事项

- 禁止把工程规范写进组件 API 页
- 禁止把配置解释散落到规范页
- 禁止把规范要求写进配置学习页
- 禁止把未发布目录当作发布库节点
- 禁止编造当前仓库不存在的导出、组件、指令、属性或脚本
- 禁止把 `components` 写成纯介绍文档
- 禁止保留已经完成迁移的旧根层文档
