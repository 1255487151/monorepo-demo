# Monorepo 项目总规范

## 1. 项目定位与适用范围

本文件是当前 monorepo 的项目总规范入口，用于统一仓库级工程规则，不替代源码实现，也不替代工具链配置本身。

本规范适用于以下目录与行为：

- `packages/components`
- `packages/directives`
- `packages/utils`
- `scripts`
- `docs`
- 根目录配置、提交流程、发布流程

本规范的目标是统一以下内容：

- 目录边界
- 代码风格
- 样式约束
- 类型与导入约束
- 组件 / 指令 / 工具库开发约束
- 构建与入口约束
- 文档协作约束
- Git / 提交 / 发布约束
- 生成物与忽略规则
- 测试与验收口径

## 2. 规则优先级与文档分工

### 2.1 规则优先级

规则优先级固定如下：

1. 仓库实际工具链配置与源码实现
2. 本文件
3. `docs/SKILL.md` 与 `docs/specs/*`
4. 包 README 与其他说明文档

若规则冲突：

- 以当前仓库实际配置和源码为准
- 再按本文件收口
- 文档细节问题交由对应专题规范处理

### 2.2 文档分工

- 根 `SKILL.md`：项目总规范，只定义项目级规则
- `docs/SKILL.md`：VitePress 文档站写法规范
- `docs/specs/*`：专题实现规范
- 包 README：包的使用说明，不承担项目总规范职责

禁止事项：

- 禁止在根 `SKILL.md` 中重复抄写 VitePress 页面写法细节
- 禁止在包 README 中承接项目总规范职责
- 禁止把专题规范全文内嵌到根规范中

## 3. 包与目录职责边界

### 3.1 `packages/components`

`components` 是基于 Element Plus 的 Vue 3 组件库，当前支持：

- 插件入口
- 纯组件入口
- 样式入口
- resolver
- 类型导出

新增组件时必须同步补齐：

- 组件源码
- 导出入口
- 样式入口
- resolver 映射
- 文档页

### 3.2 `packages/directives`

`directives` 是独立发布的 Vue 指令库。

新增导出时必须同步补齐：

- 指令实现
- 导出入口
- 注册方式说明
- 参数说明
- 文档页

### 3.3 `packages/utils`

`utils` 只用于承载无业务语义、可复用、纯函数式工具。

允许内容：

- 命名转换
- 路径解析
- 样式依赖计算
- 版本号解析
- 通用纯函数

禁止内容：

- 业务语义代码
- 构建插件逻辑
- 组件实例 glue code
- 明显只服务单一组件内部流程的局部逻辑

### 3.4 `scripts`

`scripts` 用于脚本编排和发布流程，不承接业务实现。

规则如下：

- 命令入口留在脚本文件中
- 可复用工具留在脚本级工具文件中
- 发布行为以 `scripts/version.ts` 为准

### 3.5 `docs`

`docs` 用于承载 VitePress 文档站，不承担运行时代码职责。

文档站的页面写法、结构和风格以 `docs/SKILL.md` 为准，不在本文件重复展开。

### 3.6 生成物目录

以下目录属于生成物或缓存，不纳入版本控制：

- `node_modules/`
- `dist/`
- `docs/.vitepress/cache/`
- `docs/.vitepress/dist/`

禁止提交文档缓存、构建产物和本地依赖目录。

## 4. 代码风格与工具链执行原则

### 4.1 工具链来源

代码风格和检查规则以当前仓库实际配置为准，包括但不限于：

- 根 `eslint.config.js`
- 子包 `eslint.config.js`
- `tsconfig.base.json`
- 根 `tsconfig.json`
- 子包 `tsconfig.json`
- Prettier 配置
- 根 `package.json` 脚本

### 4.2 执行原则

- 不额外引入外部完整风格体系作为主标准
- 所有新增代码必须优先满足当前 ESLint、TypeScript、Prettier 规则
- 若文档表述与工具链检查结果冲突，以工具链可执行结果为准

### 4.3 书写要求

- 优先使用清晰命名，不依赖注释弥补表达不清
- 类型、导出、目录分层必须与当前 monorepo 结构一致
- 避免同一问题同时引入多套风格约束

## 5. 样式规范

### 5.1 基本原则

- 必须优先使用本地 `scss`、语义化类名、组件 props、Element Plus 主题变量完成样式表达
- 可以消费 `--el-*`
- 禁止新增组件库自定义全局变量体系，如 `--xlg-*`

### 5.2 推荐做法

- 组件样式优先写在对应的本地 `scss` 文件中
- 样式配置优先通过 props、class、局部结构样式表达
- 布局和视觉规则优先通过语义化 class 收口

### 5.3 禁止做法

- 禁止通过新增 `--xlg-*` 变量扩展组件库样式能力
- 禁止把组件库样式能力设计为依赖宿主项目注入的自定义全局 CSS 变量
- 禁止在新代码中继续沿用历史私有全局样式变量模式

### 5.4 例外说明

- VitePress 文档站的暗色主题规则属于 `docs/SKILL.md` 管理
- 文档主题样式不进入组件库样式规范细节

## 6. 类型与导入规范

### 6.1 类型导入

- 类型导入必须优先使用 `import type`
- 运行时值导入与类型导入必须明确分离

### 6.2 别名与路径

- 根别名和包内别名必须以当前 `tsconfig` 为准
- 不允许在代码中自行发明未落到 `tsconfig` 的别名体系

### 6.3 导出边界

- 发布库导出边界必须与各包 `package.json` 中的 `exports` 一致
- 文档、示例和源码中的导入方式必须与真实导出一致

### 6.4 依赖边界

- 内部包之间的依赖以 `package.json` 为准
- 不允许绕过正式导出入口直接依赖其他包的内部私有路径

## 7. 方法注释与命名规范

### 7.1 强制规则

- 所有导出方法必须补充注释
- 所有内部复杂方法必须补充注释

### 7.2 可以不逐个补注释的场景

以下方法可以不强制逐个补注释，但命名必须足够清晰：

- 简单事件 handler
- 简单 getter
- 简单映射函数
- 只做一层转发的短函数

### 7.3 注释要求

注释必须尽量简洁地说明以下必要信息：

- 职责
- 输入 / 输出
- 特殊约束
- 副作用或依赖前置条件

注释语言默认使用中文。

禁止低信息量注释，例如：

- “设置变量值”
- “处理数据”
- “执行方法”

### 7.4 命名要求

- 命名必须体现职责和层级
- 不允许使用含糊缩写掩盖真实语义
- 命名不清时，优先重命名，不用注释补救

## 8. 组件开发规范

### 8.1 职责边界

`.vue` 文件应主要承载：

- 模板
- 响应式状态
- 组件实例绑定
- 事件编排
- `defineExpose`
- 必要的本地 glue code

禁止长期把明显可复用的纯函数堆积在 `.vue` 文件中。

### 8.2 可以保留在 `.vue` 中的内容

- 与当前组件实例强绑定的 handler
- 只服务当前组件的响应式编排
- 不具备跨组件复用价值的局部逻辑

### 8.3 新增组件时的强制同步项

- 补齐组件导出
- 补齐样式入口
- 补齐 resolver 映射
- 补齐相关类型
- 补齐文档页

### 8.4 接口变更约束

组件的 Props、Slots、Events、Exposes 发生变化时：

- 必须同步更新文档
- 必须同步检查样式和 resolver 是否受影响
- 必须说明需要运行的检查项

## 9. 指令开发规范

### 9.1 导出与注册

新增指令时必须同步补齐：

- 指令实现
- 导出入口
- 插件安装入口
- 注册方式说明
- 参数或绑定值说明

### 9.2 文档要求

指令文档至少要覆盖：

- 注册方式
- 绑定值
- 示例
- 行为说明
- 限制条件

### 9.3 行为边界

- 指令必须明确适用对象
- 指令若依赖浏览器 API，必须在文档和实现中说明边界

## 10. 工具函数规范

### 10.1 准入标准

放入 `utils` 的方法必须满足：

- 无业务语义
- 可复用
- 纯函数优先

### 10.2 禁止事项

- 禁止吸纳构建插件逻辑
- 禁止吸纳组件实例 glue code
- 禁止吸纳业务语义代码

### 10.3 测试要求

`utils` 若新增行为逻辑：

- 应补测试
- 若本轮未补测试，必须明确说明测试缺口

## 11. 构建与入口规范

### 11.1 `plugin` 目录职责

`plugin` 目录用于放置具备明确工程目的的插件能力，包括但不限于：

- 自定义 Vite / Rollup 插件
- 构建入口生成
- 产物重写
- 样式注入
- 组件入口导出拼装

凡是构建、产物、入口、样式副作用注入类逻辑，必须归类到 `plugin`，不得散落到 `utils`。

### 11.2 入口与产物规则

- 所有入口和产物策略以当前 `vite.config.ts` 与 `package.json exports` 为准
- `components` 的样式资产、运行时入口重写、resolver side effects 属于 `plugin/` 责任
- 新增入口时必须同时更新构建配置、导出边界和文档

## 12. 文档协作规范

### 12.1 根规范职责

- 根 `SKILL.md` 只定义项目级规则
- 不在根规范中复写 VitePress 页面写法细节

### 12.2 文档细则来源

- 文档写法细节统一引用 `docs/SKILL.md`
- 专题实现约束统一引用 `docs/specs/*`

### 12.3 同步规则

以下变化必须同步更新文档：

- 发布库新增或删除公开导出
- 组件 Props / Slots / Events / Exposes 变化
- 指令参数、行为或注册方式变化
- 工具函数签名或行为变化
- 配置入口、脚本命令、构建行为变化
- 专题规范变化

## 13. Git / 提交 / 发布规范

### 13.1 提交流程

- 根目录提交前必须通过 `lint-staged`
- 当前提交钩子依赖 `.husky/pre-commit`
- `.husky/pre-commit` 当前执行 `pnpm exec lint-staged --relative`

### 13.2 提交前检查

根级常用检查口径固定为：

- `pnpm run lint`
- `pnpm run format:check`
- `pnpm run type-check`
- `pnpm run code:check`

### 13.3 版本发布

版本发布以以下入口为准：

- `pnpm version`
- `pnpm release`
- `scripts/version.ts`

发布流程必须遵循：

- 工作区干净
- 按拓扑顺序处理包依赖
- 按影响传播更新依赖版本

### 13.4 禁止事项

- 禁止在工作区不干净时直接进行正式版本发布
- 禁止跳过文档和导出同步只修改版本号

## 14. 生成物与忽略规则

### 14.1 受 `.gitignore` 管理的生成物

当前必须忽略以下目录：

- `node_modules/`
- `dist`
- `docs/.vitepress/cache/`
- `docs/.vitepress/dist/`

### 14.2 提交边界

- 禁止提交文档缓存
- 禁止提交文档构建产物
- 禁止提交本地依赖目录
- 禁止把临时文件混入业务提交

## 15. 测试与验收规范

### 15.1 最低验收口径

修改代码时，至少应说明需要运行哪些检查。

默认优先口径为：

- Lint
- Format check
- Type check
- 必要时的包级构建或测试

### 15.2 包级要求

- `components` 改动后，应至少关注 lint、type-check、build 与文档同步
- `directives` 改动后，应至少关注 lint、type-check、build 与参数说明同步
- `utils` 改动后，应至少关注 lint、type-check、测试或测试缺口说明

### 15.3 文档改动

文档改动至少要保证：

- 路径引用有效
- 规范分工不冲突
- 不出现已删除文件的旧路径引用

## 16. 迁移原则

### 16.1 总体原则

- 现有历史实现若不完全符合本规范，视为待收敛项
- 新代码必须遵守本规范
- 旧代码在重构或修改时按本规范逐步收口

### 16.2 本轮不要求的事项

- 不批量重构现有代码以完全符合本规范
- 不同步修改工具链实现来强制执行所有规则
- 不在根规范中重复文档站细则

### 16.3 收口原则

- 规则冲突时，先对齐当前源码与工具链
- 文档冲突时，先对齐对应专题规范
- 无专题规范时，由根 `SKILL.md` 收口
